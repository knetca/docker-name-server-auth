# knet-zones

DNS zone files for K-Net authoritative name servers.

Consumed by `docker-name-server-auth` via SSH deploy key. Changes pushed to
`main` are picked up on the next poll cycle and applied to all NSD secondaries
without manual intervention.

## Repository structure

```
knet-zones/
├── manage_arpa/
│   ├── known_hosts           # canonical IP-to-hostname map for PTR generation
│   ├── make_reverse.sh       # generates PTR zones from known_hosts
│   └── out/                  # gitignored — working output directory
├── zones/                    # hand-managed zones — forward zones and delegations
│   ├── knet.ca.zone
│   ├── 192.184.64.in-addr.arpa.zone   # delegation zone — PicRiver ISP
│   └── ...
└── zones_arpa/               # generated PTR zones — committed after script run
    ├── 193.184.64.in-addr.arpa.zone
    └── ...
```

## Forward and delegation zones (`zones/`)

Hand-managed. One file per zone, `.zone` extension, standard RFC 1035 format.
Includes forward zones and any PTR zones requiring manual management (e.g.
delegation zones for /24s handed to external operators).

To add a zone:
1. Create `zones/<zonename>.zone`
2. Commit and push — NSD secondaries will pick it up on the next poll cycle

To remove a zone:
1. Delete the file
2. Commit and push

## PTR zones (`zones_arpa/`)

Generated by `make_reverse.sh` from `manage_arpa/known_hosts`. Do not edit
these files directly — changes will be overwritten on the next script run.

### Updating PTR records

```bash
cd manage_arpa
# Edit known_hosts as needed
vim known_hosts
# Regenerate and diff — only changed zones are updated
./make_reverse.sh
# Review changes
git diff ../zones_arpa/
# Commit
git add ../zones_arpa/
git commit -m "Update PTR records"
git push
```

`make_reverse.sh` diffs generated output against `zones_arpa/` and only
copies files where content has changed (excluding serial number). Serial
numbers are only bumped when records actually change.

### `known_hosts` format

```
<ip>    IN PTR    <fqdn>.
```

Example:
```
69.71.68.178        IN PTR courier.kuhkenah.ca.
69.71.68.179        IN PTR porter.kuhkenah.ca.
66.165.220.13       IN PTR tech.kuhkenah.ca.
```

`make_reverse.sh` reads the IP from the first field and writes everything
after it verbatim into the zone file. The `IN PTR` tokens are part of the
record and must be present. Trailing dot on the hostname is required.

Multiple entries for the same IP are supported — each generates a separate
PTR record.

## Workflow summary

| Change type | Action |
|-------------|--------|
| Forward zone record edit | Edit `zones/<zone>.zone`, bump serial, commit, push |
| New forward zone | Create `zones/<zone>.zone`, commit, push |
| Forward zone removed | Delete `zones/<zone>.zone`, commit, push |
| New delegation zone | Create `zones/<reverse>.zone` with NS records, commit, push |
| PTR record change | Edit `known_hosts`, run `make_reverse.sh`, commit `zones_arpa/`, push |
| New /24 added to `ipSpace` | Add to `make_reverse.sh`, edit `known_hosts`, run script, commit, push |
| /24 removed from `ipSpace` | Remove from `make_reverse.sh`, delete `zones_arpa/<zone>.zone`, run script to confirm 0 deployed, commit, push |
